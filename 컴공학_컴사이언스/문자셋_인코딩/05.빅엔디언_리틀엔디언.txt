<빅 엔디언, 리틀 엔디언>

엔디언(Endianness): 

	컴퓨터가 메모리에 다중 바이트(16비트, 32비트 등) 데이터를 저장하거나,  
	네트워크를 통해 전송할 때 바이트를 배열하는 순서를 의미
	
	CPU 아키텍쳐에 따라 
		1. 빅엔디언 (Big-Endian) : 순방향
		2. 리틀엔디언 (Little-Endian) : 역방향
	방식이 있다.
	
code unit: 데이터 처리 최소 단위

	텍스트 데이터를 인코딩할 때(UTF-8, UTF-16 등) 문자를 식별(처리)하기 위해 사용하는 최소 비트 단위입니다.
	
	개념: 하나의 코드 포인트(문자)를 표현하기 위해 8비트, 16비트, 32비트 단위 등으로 문자를 쪼개어 표현하는데, 
	이때 사용되는 기본 블록 크기를 말합니다.

	예시:
	
	UTF-8: 
		코드 유닛이 8비트(1바이트) 입니다. 
		영어는 1개의 코드 유닛으로, 한글은 3개의 코드 유닛으로 문자를 구성합니다.
		
	UTF-16: 
		코드 유닛이 16비트(2바이트) 입니다. 기본 문자는 1개의 코드 유닛으로, 이
		모지 같은 특수 문자는 2개의 코드 유닛으로 표현합니다

💡 빅엔디언 (Big-Endian)저장 방식: 
	
	큰 단위의 바이트가 앞(낮은 주소) 에 오는 방식입니다. 
	사람이 숫자를 쓰는 방식과 같습니다.
	
	0x12345678을 저장할 때: 12 34 56 78
	
	탄생 이유: 
		사람이 숫자를 왼쪽에서 오른쪽으로 읽는 방식과 일치하여 디버깅이 직관적입니다. 
		숫자의 크기를 비교할 때 앞자리부터 보면 되므로 크기 비교 연산이 빠릅니다.
	
	주요 사용: 
		IBM, Motorola, 과거의 SPARC 프로세서, 
		그리고 오늘날의 네트워크 프로토콜(TCP/IP) 표준.
		
💡 리틀엔디언 (Little-Endian)저장 방식: 
	
	작은 단위의 바이트가 앞(낮은 주소) 에 오는 방식입니다. 
	CPU가 메모리에서 데이터를 읽고 연산할 때 효율적입니다.
	
	0x12345678을 저장할 때: 78 56 34 12
	
	탄생 이유: 
		컴퓨터의 수학적 연산(덧셈, 곱셈 등)이 매우 효율적입니다. 
		
		올림수(Carry)가 발생하는 산술 연산은 하위 자릿수(낮은 주소)부터 시작하기 때문입니다. 
		또한 데이터 타입 변환(예: 32비트 정수를 16비트나 8비트로 캐스팅) 시 주소값을 변경하지 않고 
		앞부분만 잘라 쓰면 되므로 연산이 단순해집니다.
		
	주요 사용: 
		Intel, AMD, 그리고 x86/x64 기반의 현대 개인용 컴퓨터(PC) 및 모바일 기기의 ARM 프로세서.

	
	리틀엔디언 요약 : "인코딩 Code Unit 단위가 복수 바이트일 때 벌어지는 바이트 역순 해석"
		

1. 네트워크 전송 시: "빅 엔디언"이 절대적인 표준입니다.

	네트워크 세상에서는 기종이 다른 컴퓨터들이 수없이 데이터를 주고받기 때문에, 
	전송하는 바이트 순서를 반드시 통일해야 했습니다.
	
	네트워크 바이트 순서 (Network Byte Order): 
	인터넷 표준 프로토콜(TCP/IP)을 만들 때 빅 엔디언(Big-Endian)을 표준으로 삼았습니다.
	
	이유: 
	사람이 숫자를 왼쪽에서 오른쪽으로 읽는 방향(0x1234를 12, 34 순서로 읽음)과 같아서 
	프로토콜을 설계하고 디버깅하기 가장 직관적이기 때문입니다.
	
	개발자의 규칙: 
	따라서 리틀 엔디언을 쓰는 PC에서 데이터를 보내더라도, 
	네트워크로 쏠 때는 데이터를 빅 엔디언으로 변환(인코딩)해서 보내는 것이 
	암묵적인 약속이자 표준 규칙입니다. 
	(C언어의 htonl, htons 같은 함수들이 바로 이 'Host to Network' 변환을 해주는 녀석들)
	
2. 디스크 저장 시: "현재 CPU 아키텍처(시스템)"에 따라 다릅니다.

	디스크에 저장할 때는 네트워크처럼 거대한 표준이 강제되지 않으며, 
	프로그램이 실행되는 현재 시스템(CPU)의 환경을 따라가는 것이 일반적입니다.

	Intel / AMD (X86_64) 계열 PC: 
	현재 우리가 쓰는 대부분의 Windows, Linux PC는 리틀 엔디언(Little-Endian) 기반입니다. 
	따라서 디스크에 바이너리를 그냥 덤프 뜨면 리틀 엔디언 형태로 저장됩니다.

	스마트폰 및 최신 Mac (ARM 계열) / 대형 서버: 
	ARM CPU나 과거 대형 서버(SPARC 등)는 빅 엔디언을 쓰거나 둘 다 지원(Bi-Endian)합니다.
	
💡 여기서 발생하는 또 하나의 '인코딩' 약속: UTF-16과 BOM

	이 개념이 문자열과 만나면 선배님이 자바에서 자주 보셨을 재미있는 현상이 일어납니다.
	앞서 자바 내부가 UTF-16(2바이트)을 쓴다고 말씀드렸죠? 
	만약 자바 메모리에 있는 문자열을 디스크에 파일로 저장하려고 하면, 시스템이 리틀 엔디언이냐 빅 엔디언이냐에 따라 
	상위 바이트와 하위 바이트의 순서가 뒤바뀌어 저장될 위험이 있습니다.

	그래서 컴퓨터 과학자들은 파일 맨 앞에 BOM (Byte Order Mark)이라는 
	2바이트짜리 신호탄(0xFEFF)을 	숨겨놓는 약속을 만들었습니다.
	
		파일을 읽었는데 맨 앞에 0xFE 0xFF가 있으면 -> "이 파일은 빅 엔디언이야!"
		파일을 읽었는데 맨 앞에 0xFF 0xFE가 있으면 -> "이 파일은 리틀 엔디언이야!"
		
	라고 프로그램에게 힌트를 주는 것입니다.
	
	현 시스템과 다른 CPU에 파일을 보낼때는 만약 UTF-16 형식이라면 BOM 으로 인코딩해서 알려줘야함.
	(빅 엔디언인지, 리틀 엔디언인지)	
	
	(참고로 UTF-8은 1바이트 기반으로 쪼개기 때문에 엔디언 문제에서 완전히 자유롭습니다! 
	1바이트 읽기 기준이라서 빅엔디언,리틀엔디언 문제가 생기질 않습니다.
	그래서 웹과 파일 저장소에서 UTF-8이 대세가 된 것이기도 합니다. )
	
	(UTF-8 과 UTF-8 BOM 은 다른 변태 인코딩입니다. 실무에서는 대부분 UTF-8만을 씁니다.)
