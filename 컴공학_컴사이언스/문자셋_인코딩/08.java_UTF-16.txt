<JAVA 의 UTF-16 기준과 이모지>

java char 은 2바이트이다.

유니코드 문자셋으로 채택하고 문자 한개는 16비트 고정으로 설계되었다.

이모지 습격

	2000년대 후반 스마트폰 시대가 열리면서 폭발적인 사건이 터집니다. 
	사람들이 문자 메시지에 글자 대신 그림(이모지)을 쓰기 시작한 것입니다.
	유니코드 협회는 급하게 이모지 수천 개와 고대 희귀 문자들을 유니코드 사전에 추가해야 했습니다.

	현실: 유니코드 글자 수가 2바이트 방(65,536개)을 가볍게 뚫고 15만 개를 넘어가 버렸습니다.

	비극: 유니코드 번호가 U+1F600 같은 식으로 2바이트 영역을 초과해 버린 것입니다.

	이때 이미 전 세계 수천만 개의 시스템에 깔려있던 자바는 멘붕에 빠집니다. 
	"내 char 변수는 딱 2바이트(65,536) 크기 장난감 상자인데, 
	4바이트짜리 이모지 화물이 들어오면 어떡하지?!"
	

자바의 눈물겨운 땜질: 서러게이트 페어 (Surrogate Pair)

	자바는 이미 전 세계 표준이 되어버려서 이제 와서 char를 4바이트로 바꿀 수도 없었습니다. 
	(그랬다간 전 세계 자바 프로그램이 다 깨지니까요.)
	
	결국 자바 개발팀은 눈물을 머금고 '서러게이트 페어(대리인 커플)'라는 기괴한 땜질 규칙을 도입합니다.

	자바의 땜질: 
	
	"이모지 하나가 들어오면, 2바이트짜리 char 방 2개를 붙여서(총 4바이트) 꼼수로 표현하자!"

	이 땜질 때문에 자바 개발자들은 지금까지도 소름 돋는 '문자열 배달 사고'를 겪고 있습니다. 
	선배님이 지금 당장 자바 코드로 이모지 길이 테스트를 해보시면 눈으로 확인하실 수 있습니다.

		String emoji = "😀"; // 이모지 딱 1글자

		System.out.println(emoji.length()); 
		// 결과: 1이 아니라 2가 나옵니다! 자바 내부의 char 방을 2개 썼기 때문입니다.

	인간인 선배님 눈에는 분명히 1글자인데, 
	자바는 내부 UTF-16 장막 뒤에서 끙끙대며 방 2개를 쓰고 있기 때문에 
	length() == 2라고 대답하는 서글픈 버그 같은 현실이 발생한 것입니다. 
	
	만약 이 상태에서 substring(0, 1)로 글자를 반으로 쪼개버리면 이모지가 깨져서 
	정체불명의 사각형(``)이 튀어나옵니다.
	
Java 9 Compact Strings : 자바 심장의 비밀 성형수술
	
	자바 개발팀이 구글, 트위터, 아마존 같은 대기업들의 대규모 자바 서버 메모리를 분석해 봤더니 
	충격적인 사실이 밝혀졌습니다. 
	
	"자바 힙(Heap) 메모리의 무려 25%~30%를 String 객체가 차지하고 있는데, 그 문자열의 70% 이상이 알고 보니 
	알파벳과 숫자(ASCII)뿐이더라!"라는 점이었습니다.
	
	문제의 본질: 
		영문 'A'는 1바이트면 충분한데, 자바 8까지는 내부적으로 char[](2바이트) 배열을 썼기 때문에 
		무조건 0x00 0x41로 저장했습니다. 
		앞동네 0x00이 메모리의 수십 기가바이트를 무의미하게 낭비하고 있었던 것이죠.
		
	미션: 
		"기존에 개발자들이 쓰던 String 클래스의 메서드(length(), charAt(), substring())는 
		완벽하게 그대로 유지하면서, 내부 저장 공간만 반으로 줄여라!"
		
	🛠️ 자바 엔지니어들이 설계한 '1+1 바이트'의 마법
	자바 개발팀은 String 클래스의 내부 소스코드를 완전히 뜯어고쳤습니다
	
		// Java 8 이전의 심장
		public final class String {
		    private final char[] value; // 무조건 2바이트 방들의 배열
		}

		// Java 9 이후의 심장 (Compact Strings)
		public final class String {
		    private final byte[] value; // 1바이트 방들의 배열로 전격 교체!
		    private final byte coder;   // ★ 신의 한 수: 안경 플래그 (1바이트)
		}

	이제 자바 문자열은 생성되는 순간, 내부에 든 글자들을 검사해서 coder라는 1바이트짜리 
	플래그(안경)에 딱 이정표를 남깁니다.
	
	1. coder = 0 (LATIN1 안경)
		문자열에 영문, 숫자, 문장부호만 있을 때 선택됩니다.
		
		내부 byte[] 배열에 글자당 딱 1바이트씩만 촘촘하게 채워 넣습니다. 
		메모리가 과거 대비 정확히 50% 절약됩니다!
		
	2. coder = 1 (UTF16 안경)
		한글, 한자, 혹은 이모지가 단 하나라도 섞이면 발동합니다.
		
		"비상! 1바이트 방으로는 안 된다!" 하고는 과거 방식처럼 글자당 2바이트(혹은 이모지는 4바이트) 
		체제로 전환하여 byte[] 배열에 밀어 넣습니다.
		
	
🧩 선배님이 감탄하실 자바의 '지독한 하위 호환성' 수호작전

	여기서 소름 돋는 점은, 내부가 이렇게 변했어도 선배님이 작성하신 기존 자바 코드는 아무런 타격이 없다는 점입니다. 
	자바의 length() 메서드가 뒤에서 어떻게 연극을 짜고 있는지 보시면 무릎을 치실 겁니다.

		// 자바 9 이후의 실제 String.length() 내부 구현 원리
		public int length() {
		    // LATIN1(영문) 안경이면 byte 배열의 순수 방 개수가 곧 글자 수!
		    // UTF16(한글) 안경이면 byte 배열 방 개수를 2로 나눈 게 진짜 글자 수!
		    return this.coder == LATIN1 ? value.length : value.length >> 1; 
		}
	
	주석: >> 1은 2로 나누는 번개 같은 비트 연산입니다.
	
	개발자에게는 옛날 자바와 똑같은 얼굴을 보여주면서, 
	내부적으로는 coder 플래그를 보고 바이트를 1배수로 계산할지, 2배수로 계산할지 철저하게 장막 뒤에서 
	생쇼(?)를 하며 인간 중심의 논리적 단위를 수호해 준 것입니다.
