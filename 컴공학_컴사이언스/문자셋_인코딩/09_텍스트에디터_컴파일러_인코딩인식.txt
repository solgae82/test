<에디트 플러스에서 파일의 인코딩을 어떻게 알아서 열 수 있을까?>

에디트 플러스 뿐만 아니라 텍스트 에디터 모두에 해당하는 이야기이다
텍스트 에디터들은 열려는 파일이 무슨 인코딩인지 모른다.

단, 텍스트 에디터 제작자가 확률 알고리즘, 또는 바이트 패턴등을 이용해 
철저한 '우선순위 기반의 탐정 수사'를 거쳐 파일의 인코딩을 알아냅니다.

(파일안의 텍스트가 너무 적어서 추리가 불가능한 경우는 사용자가 선택하게 합니다.)

1단계 탐색: 명확한 '증거(BOM)'가 있는가?
	첫 2바이트 또는 3바이트 BOM 코드가 있다면 
	UTF16(2바이트 BOM) , 또는 UTF-8(3바이트 BOM) 이다!
	
2단계 추리: 증거가 없다면 '확률 통계(통계적 감지)'를 쓴다!
	파일 내부의 바이트들을 쭉 훑어보면서 "음, 바이트들이 늘어선 모양새가 
	UTF-8 특유의 수학적 규칙과 완벽하게 일치하네? 확률상 이건 99.9% UTF-8이다!
	
	대충 이런식 패턴으로 추리한다.
	
3단계 최후의 보루: 이것도 저것도 아니면 '사용자의 기본 설정'을 따른다!
	에디터 플러스 기본 설정에 설정된 기본 인코딩으로 열던가,
	'알 수 없는 인코딩인데 기본 인코딩으로 열까요?' 등의 메시지를 보내서 
	사용자가 선택하게 한다.
	
텍스트 파일이나, 소스 파일은 대부분 이런 식으로 열게 된다.

-------------------
java 컴파일러는 스스로 .java 소스 파일의 문자셋이 무엇인지 감지하지 못합니다
에디터 플러스의 고민과 같죠.

1. 과거의 방식 (Java 17 이하) : 
	"실행 중인 OS의 기본값을 믿는다"선배님이 처음 자바를 시작하셨던 시절
	(Java 1.4~1.8, 그리고 Java 17까지)에는 컴파일러(javac)가 
	다음과 같은 원칙으로 움직였습니다.
	
	원칙: 개발자가 아무 말도 안 하면, 현재 컴파일러가 실행 중인 운영체제(OS)의 기본 문자셋으로 
	파일을 읽어옵니다.
	
	과거 한글 윈도우 환경: 
		한글 윈도우의 시스템 기본 문자셋은 CP949(ANSI)입니다.
		따라서 메모장이나 옛날 에디터로 소스코드를 CP949로 저장하고 메모장에서 
		그냥 javac Test.java라고 치면 아무 문제 없이 컴파일이 잘 되었습니다.
		
	당시의 참사: 
		하지만 똑같은 소스 파일을 그대로 리눅스 서버로 가져가서 컴파일하면 에러가 났습니다. 
		리눅스의 시스템 기본값은 UTF-8이라서, CP949로 작성된 파일을 UTF-8 규칙으로 
		강제 해석하려다가 글자가 깨졌기 때문입니다.
2. 현재의 방식 (Java 18 이상) : 
	
	"무조건 UTF-8로 간주한다"자바가 20년이 넘어가면서 전 세계 개발자들이 
	이 OS별 디폴트 충돌 문제로 너무 많은 고통을 겪었습니다. 
	
	
	결국 자바 개발팀은 JEP 400 (UTF-8 by Default)이라는 대대적인 표준을 발표합니다.
	
	현재의 원칙: 
		Java 18 버전부터는 윈도우든, 맥이든, 리눅스든 상관없이 javac 명령어가 실행되면 
		소스 파일이 무조건 UTF-8로 인코딩되어 있다고 가정하고 파일을 읽습니다.
	
	현재의 참사: 
		만약 예전 프로젝트 소스 코드(CP949)를 최신 Java 21 환경에서 
		인코딩 옵션 없이 그대로 컴파일하면, 한글 주석이나 한글 문자열 리터럴이 있는 곳에서 
		말도 안 되는 문자가 포함되어 있다며 컴파일 에러가 발생합니다


3. 컴파일러에게 정답을 알려주는 법 (-encoding 옵션)
	
	컴파일러가 스스로 알아채지 못하기 때문에, "지정하기 전까지는 아무도 모른다"라는 대원칙에 따라 
	우리는 컴파일러에게 명시적으로 정답을 알려주어야 합니다.
	
	그래서 아래와 같이 인코딩을 명시적으로 알려줘야 컴파일 가능합니다.

	# 1. 이 소스 파일은 예전 방식(CP949)으로 적혀있으니 그렇게 읽어서 컴파일해줘!
	javac -encoding CP949 Test.java

	# 2. 이 소스 파일은 요즘 방식(UTF-8)으로 적혀있으니 그렇게 읽어서 컴파일해줘!
	javac -encoding UTF-8 Test.java
