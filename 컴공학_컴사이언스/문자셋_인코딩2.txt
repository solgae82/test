프로그래밍 코드 속에서 text = "Hello"라고 쓰는 문자열은 컴퓨터 입장에서 볼 때 
메모리 주소와 추상적인 기호의 결합일 뿐, 파일이나 네트워크로 보낼 수 있는 
'가공되지 않은 순수 바이트 데이터(바이너리)'가 아닙니다.

인코딩을 거치기 전까지는 컴퓨터 내부에서만 쓰는 메모리상의 객체(Object) 형태입니다.
그리고 언어별, 운영체제(OS)별로 이 내부 디폴트 상태와 기본 인코딩이 서로 다릅니다.

1. 주요 프로그래밍 언어별 내부 디폴트 상태

	현대 언어들은 메모리 내부에서 유니코드를 처리하지만, 
	내부적으로 선택한 기본 저장 구조는 다릅니다.
	
	파이썬 3 (Python 3)
	
		내부 디폴트: 가변 유니코드 (PEP 393 시스템)
	
		특징: 
			문자열에 포함된 글자 중 가장 큰 글자의 크기에 맞춰 메모리를 가변적으로 씁니다. 
			
			영문만 있으면 글자당 1바이트, 한글이 섞이면 2바이트, 
			이모지가 섞이면 4바이트 구조로 변합니다. 
			
			인코딩 규칙 없이 컴퓨터 메모리 최적화를 위해 자체적으로 관리합니다.
		
	자바 (Java) / 자바스크립트 (JavaScript) / C#
	
		내부 디폴트: UTF-16 기반
		
		특징: 
			메모리 내부에서 모든 문자를 기본적으로 2바이트(또는 4바이트) 크기로 다룹니다. 
			
			영문 'A'를 저장해도 메모리 공간은 2바이트를 차지합니다.
			
	고 (Go) / 러스트 (Rust)
	
		내부 디폴트: UTF-8 기반
		
		특징: 
			매우 독특하게도 내부 문자열 저장 구조 자체가 UTF-8 바이트 배열입니다. 
			
			언어 차원에서 외부 입출력에 가장 유리한 UTF-8을 기본 메모리 구조로 채택했습니다
			
2. 운영체제(OS) 환경에 따른 디폴트 충돌 (가장 흔한 오류의 원인)

	프로그램이 실행되는 운영체제에 따라서도 '기본 파일 인코딩' 환경이 달라서 개발자들을 자주 괴롭힙니다.
	
	리눅스(Linux) / 맥OS(macOS): 
		시스템 디폴트 인코딩이 UTF-8입니다. 
		파이썬에서 파일 인코딩을 지정하지 않고 저장해도 UTF-8로 잘 저장됩니다.
		
	윈도우(Windows): 시스템 디폴트가 CP949(확장 완complete형) 또는 MS949라는 과거 레거시 인코딩입니다.
	
	
	흔히 발생하는 참사 예시:
		리눅스 서버에서 생성한 UTF-8 텍스트 파일을 윈도우 메모장(과거 버전)이나 엑셀로 그냥 열면 
		한글이 ê° ë  ë ¤ 처럼 깨집니다. 
		윈도우는 자기 디폴트인 CP949 규칙으로 바이너리를 해석하려 고집하기 때문입니다
	
-------------------

결론: 프로그래머의 필수 규칙

컴퓨터 외부(파일, 네트워크, 데이터베이스)와 데이터를 주고 받을 때는 

	=======================
	"지정하기 전까지는 아무도 모른다" 
	=======================

라고 가정해야 안전합니다.

그래서 파이썬으로 파일을 다룰 때는 디폴트에 의존하지 않고 항상 인코딩을 명시하는 습관이 중요합니다.

	(코드 예시)
	
	# OS 디폴트에 의존하지 않고, 명확하게 UTF-8 바이너리로 변환하여 저장하는 규칙
	with open("file.txt", "w", encoding="utf-8") as f:
	    f.write("안녕하세요")
