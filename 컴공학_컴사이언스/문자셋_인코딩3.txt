<소스 코드가 UTF-8로 되어 있다고 해서 리터럴 문자 인코딩이 자동 UTF-8이 되진 않는다>

"내가 파일(소스 코드)을 UTF-8로 저장했으니, 그 안의 글자들도 당연히 UTF-8 바이트 상태로 실행되겠지?
NO!

컴퓨터 내부에서는 두 번의 거대한 변환이 일어납니다

1단계: 컴파일 시점 (물리 파일 -> 메모리 객체)

	1) 개발자가 하드디스크에 Test.java 파일을 UTF-8 형식으로 저장합니다. 
	이때 파일 속 "가"라는 글자는 하드디스크에 0xEA 0xB0 0x80 (3바이트)로 적혀 있습니다.
	
	2) 자바 컴파일러(javac)가 이 파일을 읽어서 .class 바이트코드로 바꿉니다.
	
	3) 프로그램이 실행(JVM 구동)되면서, 이 바이트를 읽어 자바의 내부 규칙인 UTF-16 형태의 
	메모리 객체(0xAC 0x00)로 변환하여 힙(Heap) 메모리에 올립니다.
	
	※ 핵심: 소스 파일이 UTF-8이었든, 다른 형식이었든 상관없이, 메모리에 올라온 순간 
	소스 파일의 인코딩 껍데기는 완전히 벗겨지고 '자바 전용 내부 상태(유니코드 객체)'로 통일됩니다
	
2단계: 런타임 시점 (메모리 객체 -> 외부 전송)

	이제 메모리에 있는 "가"를 파일로 쓰거나 네트워크로 전송해야 하는 순간이 옵니다.
	
	1) 자바는 메모리에 있는 0xAC 0x00을 밖으로 내보내려고 준비합니다.
	
	2) 이때 "자, 이걸 밖으로 보낼 때 어떤 규칙(인코딩)으로 가공해서 보낼까?"를 결정해야 합니다.
	
	3) 만약 소스 코드가 UTF-8 파일이었다는 사실에 안주하여 여기서 인코딩을 명시하지 않으면, 
	자바는 실행 중인 운영체제(OS)의 기본 인코딩(예: Windows의 MS949)을 가져와서 변환해 버립니다.
	
	4) 결국 UTF-8로 만들었던 소스 코드가 윈도우 서버에서 실행되면 느닷없이 
	MS949 바이너리로 출력되는 대참사가 일어나는 것입니다.
	
-------------------
결론: 왜 항상 인코딩을 명시해야 하는가?

소스 파일의 인코딩(UTF-8)은 컴파일러에게 소스 코드를 읽어달라고 할 때만 유효한 규칙입니다. 
프로그램이 실행된 이후(Runtime)의 세상과는 완전히 단절됩니다.

그렇기 때문에 아래 코드처럼 명시적 인코딩을 지정해서, 과거 소스 파일의 규칙을 이어받는 게 아니라 
"메모리에 추상적으로 존재하는 유니코드를 지금 이 순간 새로 UTF-8 바이너리로 굽겠다" 라고 
선언하는 독립적인 명령을 해줘야한다.

(인코딩 명시 예제)

// "메모리에 있는 String을 지금 '새로' UTF-8 바이트로 쪼개줘"라는 의미
byte[] bytes = text.getBytes(StandardCharsets.UTF_8);

// "지금 들어오는 바이너리는 UTF-8 규칙으로 채워져 있으니, 자바 내부 유니코드로 조립해줘"라는 의미
BufferedReader br = new BufferedReader(new InputStreamReader(
	socket.getInputStream(), StandardCharsets.UTF_8));
