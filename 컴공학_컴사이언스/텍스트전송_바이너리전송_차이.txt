
<텍스트 전송과, 바이너리 전송의 차이>

---------------

"텍스트 파일 속 'ab'는 무조건 이진수 01100001 01100010이다"라고 
정해진 것이 아닙니다.

'ab'라는 문자를 어떤 인코딩(UTF-8, UTF-16, UTF-32 등) 규칙의 필터로 걸러서 
컴퓨터에게 넘겨주느냐에 따라 메모리에 꽂히는 이진수 알맹이는 완전히 다른 모양이 됩니다.

그래서 개발자들은 항상 "변환할 때 인코딩 방식을 명시"해 주어야 합니다.

심지어 컴퓨터가 데이터를 앞에서부터 읽느냐(Big Endian), 
뒤에서부터 읽느냐(Little Endian)에 따라 앞뒤 순서가 뒤바뀌어 
01100001 00000000 ...처럼 저장되기도 합니다.

1. 자바 코드로 보는 변환의 실체

	자바의 String은 내부적으로 문자를 UTF-16 구조로 다룹니다. 
	이것을 파일로 저장하거나 네트워크로 보낼 때 우리는 습관적으로 인코딩을 지정합니다.

	String text = "ab";

	// 1. UTF-8 바이너리로 변환 (2바이트)
	byte[] utf8Bytes = text.getBytes(StandardCharsets.UTF_8); 

	// 2. UTF-16 바이너리로 변환 (4바이트 + BOM 추가 시 더 늘어남)
	byte[] utf16Bytes = text.getBytes(StandardCharsets.UTF_16); 

	위의 두 byte[] 배열은 내부적으로 완전히 다른 이진수(바이너리)를 담고 있습니다.
	즉, 자바에서 text.getBytes()를 호출하는 그 순간이 바로 
	"텍스트를 바이너리로 변환하는 순간"입니다. 

	메모리 속에 추상적으로 존재하던 문자열 객체를, 물리적인 디스크나 네트워크에 꽂아 넣을 수 있는 
	'실제 바이트 덩어리'로 쪼개는 작업인 거죠.

2. 엔디언(Endian)까지 고려하면 완전한 타인

	리틀 엔디언(Intel 계열 CPU)과 빅 엔디언(네트워크 표준 등)의 차이까지 들어가면, 
	똑같은 'ab'가 메모리 버퍼 상에서 97, 0, 98, 0이 되기도 하고 
	0, 97, 0, 98이 되기도 합니다.

	실체(0과 1)가 아예 바뀌어 버리니, "같은 데이터인데 왜 변환한다고 하지?"라는 의문이 
	"아, 다른 형태로 재배치하니까 변환이 맞구나!"로 풀리게 되는 것입니다

-------------------

1. 텍스트 형식의 전송 (Text Protocol)
	
	데이터를 무조건 "사람이 읽을 수 있는 문자열"로 바꾼 뒤, 가공해서 보내는 방식. 
	대표적인 예로 HTTP/1.1, JSON, XML, REST API가 있음.

	동작 방식: 만약 숫자 12345를 보낸다면, 
	이 숫자를 문자열 "12345"로 변환(인코딩)하여 보냄.
	
	실제 전송 내용: 문자 '1', '2', '3', '4', '5'의 ASCII/UTF-8 바이트인 
	49, 50, 51, 52, 53이 전송. (총 5바이트 사용)
	
	특징:Wireshark 같은 툴로 네트워크 패킷을 떠보면 
	데이터가 텍스트로 그대로 읽혀서 디버깅이 아주 편해짐.
	
	하지만 데이터 용량이 커지고, 
	받는 쪽에서 숫자로 파싱(Integer.parseInt())해야 하므로 오버헤드 발생.
	
2. 바이너리 형식의 전송 (Binary Protocol)

	데이터를 문자열로 바꾸지 않고, "컴퓨터 메모리에 있는 이진수 값(바이트) 그대로" 
	전송선로에 밀어 넣는 방식. 
	
	대표적으로 HTTP/2, HTTP/3, gRPC, Protobuf, 웹소켓 바이너리 프레임 등.

	동작 방식: 
		숫자 12345를 보낸다면, 자바의 4바이트 int형 메모리 구조 그대로 보냄.
	
	실제 전송 내용: 
		12345는 이진수로 0x00003039이므로, 
		바이트 배열로 0, 0, 48, 57이 전송. (총 4바이트 사용)
	
	특징: 
		패킷을 가로채서 열어보면 사람이 읽을 수 없는 깨진 글자(??09 같은 형태)로 보임.
		데이터 크기가 최소화되고 문자열 파싱 과정이 없기 때문에 속도가 압도적으로 빠르고 효율적.

-------------------
		
결론
	모든 컴퓨터 데이터는 결국 0과 1이라는 실체로 수렴하지만,
	
	텍스트 전송은 "인간 중심적인 규격(문자열 포장지)"을 씌워서 다루는 방식이고,
	바이너리 전송은 "컴퓨터 중심적인 규격(메모리 바이트 그대로)" 효율성을 극대화해 다루는 방식
	
	텍스트 형식 전송: 
		"야, 서로 알아보기 편하게 전부 문자열(String)로 통일해서 보내자!" 
		(JSON, HTTP/1.1)
		
	바이너리 형식 전송: 
		"문자열로 바꾸면 용량만 커지잖아. 
		그냥 메모리에 있는 바이트 배열(byte[]) 그대로 쏴!" 
		(gRPC, HTTP/2)
