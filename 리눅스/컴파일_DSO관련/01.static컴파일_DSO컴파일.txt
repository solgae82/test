리눅스 환경에서 Static(정적) 컴파일과 DSO(Dynamic Shared Object, 동적) 컴파일은 
라이브러리나 모듈을 실행 파일에 연결(Linking)하는 방식의 차이입니다. 

주로 Apache 웹 서버 모듈 설치나 C/C++ 프로그램 빌드에서 이 두 방식을 다룬다.
핵심적인 차이는 라이브러리를 실행 파일에 포함시키느냐(Static), 필요할 때 불러오느냐(DSO)입니다.

1. Static 컴파일 (정적 컴파일/링킹)
	컴파일 시점에 필요한 모든 라이브러리나 모듈 코드를 실행 파일 내부로 가져와서 하나의 큰 바이너리 파일을 만드는 방식입니다. 
	
	특징:
		독립성: 라이브러리가 포함되어 있어 다른 환경에서도 라이브러리 의존성 문제없이 실행 가능합니다.
		실행 속도: 프로그램 시작 시 라이브러리를 찾거나 로드하는 과정이 없어 조금 더 빠릅니다.
		파일 크기: 라이브러리가 포함되므로 실행 파일의 크기가 매우 커집니다.
		업데이트: 라이브러리가 변경되면 프로그램을 다시 컴파일하고 다시 배포해야 합니다.
	
	장점: 라이브러리 버전 불일치 문제(Dependency Hell)가 없음.
	단점: 메모리/디스크 공간을 많이 사용하며, 모듈 추가/삭제 시 전체 재컴파일 필요. 


2. DSO 컴파일 (Dynamic Shared Object, 동적 컴파일)
	
	컴파일 시점에는 라이브러리의 정보만 넣어두고, 프로그램이 실행(Runtime)될 때 
	외부 공유 라이브러리(.so 파일)를 불러오는 방식입니다. 
	
	특징:
		의존성: 라이브러리가 별도로 존재해야 실행 가능합니다.
		실행 속도: 라이브러리를 로드하는 과정 때문에 정적 컴파일보다 아주 미세하게 느릴 수 있습니다.
		파일 크기: 실행 파일 크기가 작습니다.
		업데이트: 해당 라이브러리(.so)만 교체하면 프로그램 재컴파일 없이 기능 수정이 가능합니다.

	장점: 메모리/디스크 공간 효율적, 모듈 유지보수 용이(아파치 모듈 등).
	단점: 라이브러리 파일이 없거나 버전이 다르면 실행되지 않음. 
	
	
3. Static vs DSO 요약 비교
	
	구분 		Static (정적)			DSO (동적)
	--------------------------------------------------------------------------
	라이브러리 포함	컴파일 시 실행파일 내부에 포함		실행 시점에 외부에서 로드 (.so)
	
	실행 파일 크기	큼				작음
	
	라이브러리 변경	전체 재컴파일 필요			라이브러리 파일만 교체 (.so만 변경)
	
	실행 속도		약간 빠름				약간 느림
	
	메모리사용		높음 (여러 프로그램시)		낮음 (라이브러리 공유)
	
	적합한 경우		독립적 실행이 필요한 도구, 임베디드	웹 서버 모듈, 유지보수가 중요한 경우


4. 언제 무엇을 사용해야 하나요?
	
	DSO 컴파일 추천: 
		Apache 웹 서버 모듈(mod_so), 유지보수가 자주 필요한 애플리케이션. 
		라이브러리를 여러 프로그램이 공유해야 할 때.
	Static 컴파일 추천: 
		배포 환경이 다양하여 환경 변수를 맞추기 어려운 경우, 
		라이브러리 의존성 문제를 피하고 싶을 때, 단일 파일로 배포해야 할 때.
	
	최근 시스템은 하드웨어 성능이 좋아져서, 유지보수와 효율성이 뛰어난 DSO 방식이 선호되는 추세입니다.
