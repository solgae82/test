
LD_PRELOAD는 리눅스 동적 링커(ld.so)가 프로그램을 실행할 때, 
그 어떤 공유 라이브러리(.so)보다 최우선으로 특정 라이브러리를 강제 로드하도록 만드는 환경 변수입니다 

이 변수에 지정된 라이브러리는 RPATH, RUNPATH, LD_LIBRARY_PATH는 물론, 
표준 C 라이브러리(libc.so)보다도 먼저 프로세스 메모리에 적재됩니다.

1. 작동 원리: 함수 가로채기 (Hooking)

	프로그램이 어떤 함수(예: open, malloc)를 호출할 때, 
	동적 링커는 메모리에 먼저 로드된 라이브러리부터 순서대로 함수 이름을 검색합니다.

	LD_PRELOAD에 등록된 라이브러리가 가장 먼저 로드되므로, 
	기존 표준 라이브러리에 정의된 함수와 동일한 이름의 함수를 만들어 두면 
	원본 함수를 가로채서(Hooking) 내 코드를 실행할 수 있습니다.

	[일반적인 흐름]
		1) 프로그램 실행
		2) libc.so 로드
		3) 표준 malloc() 호출

	[LD_PRELOAD 사용 시 흐름]
		1) 프로그램 실행
		2) custom.so 강제 선행 로드
		3) libc.so 로드
		4) 변형된 custom_malloc() 호출

2. 주요 활용 사례

	디버깅 및 프로파일링: 
		malloc과 free를 가로채서 메모리 누수(Leak)를 추적하거나 
		메모리 할당량을 실시간으로 측정할 수 있습니다.
	
	기능 수정 및 모니터링: 
		소스 코드가 없는 상용 프로그램이나 이미 컴파일된 바이너리를 수정하지 않고, 
		특정 파일에 접근(open)하거나 네트워크 연결(connect)을 시도할 때 
		로그를 남기거나 차단할 수 있습니다.
		
	특수 기능 라이브러리 결합: 
		성능이 더 뛰어난 메모리 할당자(예: Google의 tcmalloc, 페이스북의 jemalloc)를 
		기존 프로그램의 재컴파일 없이 적용할 때 사용합니다.

3. 실제 사용 예시 
	
	(명령어 표준 패턴)프로그램 실행 직전에 환경 변수를 일회성으로 선언하여 병합 구동합니다.

	# 1. tcmalloc 라이브러리를 임시로 선행 로드하여 프로그램 성능 테스트
	LD_PRELOAD="/usr/lib/libtcmalloc.so" ./my_program

	# 2. 직접 만든 후킹 라이브러리(my_hook.so)를 적용하여 실행
	LD_PRELOAD="./my_hook.so" ./my_program
	
4. 보안 제약 사항 

	(Setuid 바이너리)LD_PRELOAD는 시스템 전체를 해킹할 수 있는 
	위험성이 있으므로 강력한 보안 규칙이 적용됩니다.

	Setuid/Setgid 권한 무효화: 
		실행 파일에 setuid 권한(루트 권한으로 실행 등)이 걸려 있는 경우, 
		보안을 위해 동적 링커는 LD_PRELOAD 환경 변수를 통째로 무시합니다.
		
	예외: 
		단, /etc/ld.so.preload 파일에 라이브러리 경로를 직접 등록하면 
		setuid 파일에도 적용할 수 있지만, 
		이는 시스템 관리자(root) 권한이 있어야만 수정이 가능합니다.
