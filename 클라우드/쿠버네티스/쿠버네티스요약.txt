[그림으과 실습으로 배우는 도커 & 쿠버네티스] 위키북스 책 중 쿠버네티스 요약

쿠버네티스는 컨테이너 오케스트레이션 도구의 일종이다.

쿠버네티스는 여러 대의 컨테이너가 여러 대의 물리 서버에 걸쳐 실행되는 것을 전제로 한다.

쿠버네티스구조

	클러스터 
		마스터 노드(요즘엔 컨트롤 플레인이라고 한다)
			워커 노드에서 실행되는 컨테이너를 관리하는 역할을 한다
			컨트롤 플레인 : 클러스터 관리
				- API 서버: 
					kube-apiserver:
						외부와 통신하는 프로세스,kubectl로 명령를 전달받아 실행
				- 컨트롤 매니저: (다양하게 많다)
					kube-controller-manager:
						컨트롤러를 통합 관리, 실행
					cloud-controller-manager: 
						클라우드 서비스와 연동해 서비스 생성
				- 스케쥴러: 
					kube-scheduler: 파드를 워커 노드에 할당한다
				- etcd: 클러스터 관련 정보 전반을 관리하는 데이터베이스
		워커 노드('노드'라고도 한다)
			컨테이너가 동작해야하는 곳
			컨테이너 엔진이 설치돼야 한다
			
			kube-let: 
				- 마스터 노드에 있는 kube-scheduler와 연동하면 워커 노드에 파드를 배치하고 실행.
				- 실행 중인 파드의 상태를 정기적으로 모니터링하며 kube-scheduler에 통지.
			kube-proxy: 
				네트워크 통신의 라우팅 역할.  
				network rule 을 관리하여 쿠버네티스 cluster의 내부 
				또는 외부의 network 연결을 도와주는 역할을 한다.
			
			컨테이너 런타임: 도커, containerd 등..
			
쿠버네티스 철학 : 
	- 정의된 바람직한 상태(.yaml)를 자동으로 생성, 모니터링, 유지
	- 유지되는 상태를 변경하고 싶다면 정의된 바람직한 상태(.yml)를 변경한다.
		(.yaml을 통하지 않고 중간에 사람이 개입해서 컨테이너를 추가/삭제/변경 하면 안된다)
		
		
쿠버네티스 중요 리소스(오브젝트)

	- 파드 (pod)
		가장 작은 배포 단위
		파드는 컨테이너+volume  묶은 것으로 이해하면 편하다
		기본적으로 파드1개 = 컨테이너 1개 이지만, 컨테이너가 여러개인 파드가 있을 수도 있다
		
	- 서비스 (service)				
		역할: 
			내부/외부 클라이언트가 동적으로 변하는 파드 IP 대신 안정적으로 서비스에 접근할 수 있게 합니다.
			파드의 묶음(Label Selector로 매핑)에 대해 고정된 네트워크 엔드포인트(IP포트)를 제공.
			파드는 자주 변경되지만 서비스는 고정된 IP를 통해 클라이언트에게 안정적인 서비스 주소를 제공
			
		구조: Label Selector를 사용해 특정 디플로이먼트로 생성된 파드들을 묶어 연결합니다.
		타입: ClusterIP(내부), NodePort(외부 노드 포트), LoadBalancer(클라우드 로드밸런서) 등이 있습니다.
		
	- 레플리카세트(replicaset)
		파드 수를 관리하는 반장
		레플리카세트가 관리하는 동일한 파드를 'replica' 라고 한다
		단독으로 쓰이기 보다는, 디플로이먼트와 함께 쓰인다.
		
	- 디플로이먼트 (Deployment)
		파드의 디플로이(배포)를 관리한다
		레플리카세트 위에 있는 상사

	파드를 관리하는 두 주체
		- 서비스 요청 관리 : 서비스 , cluster IP
		- 파드 수 관리: 레플리카세트, 디플로이먼트
		
		
	리소스 핵심 구조 요약:
		사용자/네트워크 -> [서비스(Service)] -> [디플로이먼트(Deployment)] -> [레플리카셋] -> [파드(Pod)들]

	위의 4개 중요 리소스 말고도 모두 합해 50여종 (volume,label,namespace..등)이 넘는다고 한다.
	일단 위 중요 4개 리소스는 기억하자

로드 밸런서 or 인그레스(Ingress)
	워커 노드에 요청을 분배하는 역할
	
	인그레스(Ingress)
		클러스터 내의 서비스에 대한 외부 접근을 관리하는 API 오브젝트, 일반적으로  HTTP 를 관리.
		부하 분산, SSL종료, 명칭 기반 가상 호스팅을 제공

--------- 무작정 생성 실습하기
	
1. 윈도우즈 도커.desktop에서 Kubenetes 메뉴로 쿠버네티스 설치 실행

2. D:\kube\deployment.yaml (디플로이먼트 작성)

	apiVersion: apps/v1
	kind: Deployment
	metadata: 
	  name: my-deployment
	spec:
	  selector: 
	    matchLabels: 
	      app: my-pod-lable
	  replicas: 3
	  template: 
	    metadata: 
	      labels:
		app: my-pod-lable
	    spec:
	      containers: 
	      - name: my-container
		image: httpd
		ports:
		- containerPort: 80
	
3. D:\kube\service.yaml (서비스 작성)

	apiVersion: v1
	kind: Service
	metadata:
	  name: my-service
	spec:
	  type: NodePort
	  ports: 
	  - port: 8099
	    targetPort: 80
	    protocol: TCP
	    nodePort: 30080
	  selector: 
	    app: my-pod-lable
    
4. 디플로이먼트(deployment) 생성
	
	$ cd D:\kube 
	
	# deployment.yaml 적용
	$ kubectl apply -f .\deployment.yaml
	deployment.apps/my-deployment created
    
	# 생성된 파드 확인
	#(replicas : 3 이므로 3개가 생성된게 확인된다)
	
	$ kubectl get pods
	NAME                             READY   STATUS    RESTARTS   AGE
	my-deployment-5fcb99845d-4ttfp   1/1     Running   0          17m
	my-deployment-5fcb99845d-mvr22   1/1     Running   0          17m
	my-deployment-5fcb99845d-twc44   1/1     Running   0          17m
	
5. 서비스(service) 생성

	$ cd D:\kube
	
	# 서비스 생성
	$ kubectl apply -f .\service.yaml
	service/my-service created
	
	# 생성된 서비스 확인
	$ kubectl get services
	NAME         TYPE        CLUSTER-IP     EXTERNAL-IP   PORT(S)          AGE
	kubernetes   ClusterIP   10.96.0.1      <none>        443/TCP          16h
	my-service   NodePort    10.109.52.90   <none>        8099:30080/TCP   104s
	
6. http://localhost:30080/ 확인(아파치 실행되었는지)

--------- 파드 갯수 늘리기 실습

1. deployment.yaml 의 replicas 수 변경
	
	..
	  replicas: 5 # <=여기, 3에서 5로 변경
	..

--------- 파드, 디플로이먼트, 서비스 삭제

1. 파드 모두 삭제하기

	# deployment.yaml 의 replicas 수 '0' 으로 변경 
	
	..
	  replicas: 0 # <= 파드 수를 0으로 변경, 파드 모두 삭제
	..
	
	# 변경 적용
	$ kubectl apply -f .\deployment.yaml
	deployment.apps/my-deployment configured
	
	# 파드 확인
	$ kubectl get pods
	No resources found in default namespace
	
2. 디플로이먼트 삭제
	
	# deployment 확인
	$ kubectl get deployment
	NAME            READY   UP-TO-DATE   AVAILABLE   AGE
	my-deployment   0/0     0            0           54m

	# deployment 삭제
	$ kubectl delete -f .\deployment.yaml
	deployment.apps "my-deployment" deleted from default namespace

	# deployment 삭제 되었는지 확인
	$ kubectl get deployment
	No resources found in default namespace
	
3. 서비스 삭제

	# 현재 서비스 확인 (my-service 가 생성한 서비스)
	$ kubectl get services
	NAME         TYPE        CLUSTER-IP     EXTERNAL-IP   PORT(S)          AGE
	kubernetes   ClusterIP   10.96.0.1      <none>        443/TCP          17h
	my-service   NodePort    10.109.52.90   <none>        8099:30080/TCP   55m

	# 서비스 삭제
	$ kubectl delete -f .\service.yaml
	service "my-service" deleted from default namespace

	# 서비스 삭제 확인
	$ kubectl get services
