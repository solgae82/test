Docker Compose 환경 변수는 나중에 선언한 것이 이전에 설정된 값을 덮어씁니다 (Override).
FLASK_ENV뿐만 아니라 데이터베이스 비밀번호, 포트 번호 등 모든 환경 변수 옵션이 동일하게 작동합니다.
이와 관련된 핵심 작동 규칙을 스캔하기 좋게 요약해 드립니다.

1. 환경 변수 우선순위 (덮어쓰기 규칙)

	도커는 같은 이름의 변수가 여러 곳에 있으면 아래 순서대로 
	가장 번호가 높은(마지막) 값을 최종 적용합니다.
	
	1) Dockerfile 내부의 ENV 설정 (가장 낮음)
	2) 프로젝트 폴더의 .env 파일에 적힌 값
	3) docker-compose.yml 파일의 environment: 항목 
	4) 컨테이너를 켤 때 터미널에 직접 입력한 값 (가장 높음)

2. 다른 옵션들도 마찬가지인가요?
	네, docker-compose.yml에 적는 대부분의 실행 옵션들이 동일하게 덮어쓰기 형태로 적용됩니다.

	- 실행 명령어(CMD): 
		Dockerfile에 CMD ["python", "app.py"]라고 구웠어도, 
		Compose 파일에 command: gunicorn server:app라고 적으면 
		Compose가 지정한 명령어로 대체되어 실행됩니다.

	- 포트(Ports): 
		Dockerfile의 EXPOSE 5000은 단순 안내판일 뿐이며, 
		실제 문을 열고 닫는 것은 Compose의 ports: - "5000:5000" 설정이 최종 결정합니다.

3. 실무에서 이 특성을 활용하는 방법이 구조 덕분에 우리는 이미지를 딱 한 번만 빌드하고도 환경에 맞춰 변신시킬 수 있습니다.

	- 이미지는 solgae/my-flask-app:1.0 하나만 만들어 둡니다.
	- 내 컴퓨터의 Compose 파일에는 - FLASK_ENV=development를 주어 개발 모드로 켭니다.
	- 맥미니 우분투의 Compose 파일에는 - FLASK_ENV=production을 주어 운영 모드로 켭니다.


결국 이미지를 새로 구울 필요 없이 docker-compose.yml 파일의 환경 변수 한 줄만 바꾸면 
상황에 맞게 덮어쓰기가 완료되므로 매우 편리하고 안전합니다.

실무에서는 이 특성을 활용해 보안과 유연성을 동시에 챙깁니다.

보안 강화: 
	데이터베이스 비밀번호나 API 키 같은 민감한 정보를 이미지 안에 저장(빌드)하면 해킹 위험이 있습니다. 
	대신 이미지는 빈 껍데기로 구워두고, 
	실제 리눅스 서버의 docker-compose.yml 환경변수로 비밀번호를 주입(덮어쓰기)하여 안전하게 운영합니다.

불변의 이미지 (Immutable Image): 
	개발용 이미지, 테스트용 이미지, 배포용 이미지를 따로 만들지 않습니다. 
	알맹이는 완전히 똑같은 이미지 하나를 쓰고, 오직 겉싸개(Compose 파일)만 바꾸어 상황에 맞게 변신시킵니다.
